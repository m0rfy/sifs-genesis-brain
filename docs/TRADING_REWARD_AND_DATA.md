# Reward и данные для торговой среды (§6 и §9 плана, D.3)

Документ задаёт источник reward, нормировку и способ доставки в мозг для мини-торговой среды, а также источник данных и правила train/val. Реализация — при фазах D.3.

---

## §6. Reward для торговой среды

### Выбор варианта

| Вариант | Описание |
|--------|-----------|
| PnL за шаг | Прибыль/убыток по открытой позиции за один бар (или по закрытию шага). |
| Sharpe за окно | Sharpe ratio за скользящее окно; реже обновляется. |
| Штраф за drawdown | Дополнительный отрицательный reward при просадке от пика. |
| **Отложенный reward** | Начислять при закрытии сделки; размазывать по шагам (reward shaping) или отдельный канал с задержкой. |

**Зафиксировано для D.3 (минимум):** **PnL за шаг** — разница стоимости позиции (или PnL по закрытию шага) в единицах, нормированная к диапазону, совместимому с дофамином. При появлении реального пайпа с закрытием сделок — добавить **отложенный reward при закрытии** (вариант (a): размазывать по шагам сделки).

### Нормировка

- Целевой диапазон для поля `reward` в контракте мозга: **float, например [-1, 1]** или [0, 1] при масштабировании (как в CartPole). Формула: `reward_normalized = clip((pnl - pnl_min) / (pnl_max - pnl_min + eps), 0, 1)` или симметричная для знака.
- Параметры (pnl_min, pnl_max или масштаб) — в конфиге торговой среды или в BRAIN_CONTRACT §6.2 при уточнении.

### Доставка в мозг

- Тот же контракт, что и для CartPole: поле `"reward"` в JSON (stdio/HTTP) по BRAIN_CONTRACT §3.1. Мозг использует reward в Night Phase для модуляции пластичности. Клиент торговой среды (sifs_ft или отдельный runner) передаёт reward за предыдущий шаг вместе с наблюдением.

## §9. Данные для торговли

### Источник

- **Основной:** OHLCV в формате `.feather` в каталоге `sifs_ft/data/binance/` (или аналог при смене биржи). Загрузка: скрипты из sifs_ft (см. BACKTEST_RESULTS.md, download_backtest_data.ps1).
- **Таймфреймы и пары:** по конфигу стратегии sifs_ft; для обучения мозга при D.3 зафиксировать один таймфрейм (например 15m) и набор пар (например BTC/USDT, ETH/USDT) в конфиге среды.

### Временной диапазон и разбиение

- **Train/val split по времени** (без перемешивания по времени): например train до даты T, val от T до T_end. Даты начала/конца зафиксировать в конфиге или в этом документе при первом запуске D.3.
- **Без look-ahead:** при формировании наблюдения на времени t использовать только данные до t (включительно). Индикаторы (SIFS, EMA и т.д.) считаются только по прошлым барам.

### Overfitting и тестирование

- Тестировать на **out-of-sample** периоде (после val или отдельный test segment). Не использовать будущие данные при обучении и при выборе гиперпараметров по val.
- Кратко описать в отчёте или в конфиге: диапазоны train/val/test, пары, таймфрейм.
