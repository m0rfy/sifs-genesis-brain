# Механизм против катастрофического забывания (D.2)

Один мозг — несколько сред (например CartPole + торговый мини). Без явной схемы обучение на одной задаче ухудшает другую. Ниже зафиксирован **выбранный минимальный механизм** и критерий деградации.

---

## Варианты (из плана)

| Вариант | Описание |
|--------|----------|
| (a) Task-conditioned gating | Вход задачи/контекста в сеть; разные «ворота» по задаче. |
| (b) Отдельные чекпоинты на задачу с переключением | Один набор весов на задачу; при смене среды загружается чекпоинт этой задачи. |
| (c) Rehearsal/replay буфер | Хранить примеры из предыдущих задач и подмешивать при обучении. |
| (d) Явное разделение зон | Часть нейронов/шардов закреплена за задачей. |

---

## Выбор: **(b) Отдельные чекпоинты на задачу с переключением**

**Обоснование:** минимальная реализация без изменения архитектуры мозга. Каждая среда (CartPole, торговый мини) имеет свой чекпоинт весов; при переключении среды оркестратор загружает соответствующий чекпоинт. Обучение на задаче A не перезаписывает веса задачи B.

**Ограничения:** нет совместного обучения на нескольких задачах «одним мозгом»; при большом числе задач растёт число чекпоинтов. Для 2–3 сред этого достаточно; при росте числа задач рассмотреть (a) или (c).

---

## Критерий деградации

- **Определение:** при обучении/инференсе на задаче B метрика на задаче A не должна падать ниже порога X (если периодически проверяем A).
- **Практика при схеме (b):** при переключении на задачу A загружаем чекпоинт A; метрика A не страдает от обучения на B, так как веса A не перезаписываются. Деградация возможна только внутри одной задачи (перезапись чекпоинта A при обучении на A). Тогда критерий: при сохранении нового чекпоинта A сравнивать метрику с предыдущим чекпоинтом A; при падении ниже X — не перезаписывать, откатиться (см. [CHECKPOINTING_WEIGHTS.md](CHECKPOINTING_WEIGHTS.md)).
- **Порог X:** задаётся в конфиге среды или в плане (например для CartPole: медиана score ≥ 50 как минимальный порог «не деградировали»).
