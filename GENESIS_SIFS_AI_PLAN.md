# План: Genesis + SIFS → полноценный ИИ

Стратегический и инженерный документ: движок **Genesis** (Rust/CUDA) + теория **SIFS** → агент с проверяемыми критериями готовности. Версия 2 — с учётом критических пробелов, рисков и операционализации «done».

---

## 1. Визия и операциональное определение «полноценный ИИ»

**Цель:** мозг с физически обоснованной архитектурой (SIFS + GLIF + пластичность), используемый в симуляциях и в продакшене.

**«Полноценный ИИ» здесь задаётся критериями завершения фаз, а не абстрактной формулировкой:**

| Уровень | Критерий «done» |
|---------|------------------|
| **Фаза A** | Константы синхронны, регрессия I_SIFS в CI зелёная, гипотеза SIFS→GLIF верифицирована или опровергнута (см. §7). |
| **Фаза B** | CartPole: медиана score ≥71 за 50 эпизодов с фиксированным сидом; A/B с/без I_SIFS задокументирован. |
| **Фаза C** | Один контракт наблюдение→действие; мозг доступен по HTTP и/или UDP; fallback при недоступности описан; latency budget зафиксирован. |
| **Фаза D** | Один развёрнутый мозг обслуживает ≥2 среды (например CartPole + торговый мини) без деградации ниже порога (см. §8 катастрофическое забывание); чекпоинты весов и версионирование работают. |

Без выполнения критериев фаза не считается завершённой. CartPole 71+ — не «полноценный ИИ», а **контрольная точка** фазы B; итоговый «полноценный ИИ» — выполнение критериев фазы D + документированная связь теория ↔ реализация. **Торговля (D.3)** — одна из сред для того же мозга (обучение, настройка, продакшен), не замена цели «полноценный агент».

- **Выбрано:** pre-commit hook (Windows, PowerShell) + при необходимости local script. Альтернативы: GitHub Actions (если есть .github/ в репо Genesis), Windows-батник как «local CI».

Без выбора CI A.1 не «заблокирован», но регрессия остаётся ручной; при первом рефакторинге константы могут разъехаться.

## 3. Центральная гипотеза и риски (верификация в фазе A)

**Гипотеза:** пороги V_th(n)=V₀/φⁿ и ток I_SIFS (Σ W(n)·(price−EMA_n)), выведенные из фрактального анализа рынков (SIFS), переносятся в GLIF-нейроны как **мультимасштабная модуляция порога/тока** и улучшают обучение или обобщение по сравнению с фиксированным порогом.

**Почему это не очевидно:** на рынках «price» — скаляр по времени; в нейросети «price» мы подменили напряжением нейрона. Связь масштабов рынка (таймфреймы) с S-уровнями 0..9 и с одной нейронной популяцией нигде не обоснована формально.

**Что сделать в фазе A (обязательно):**

- **A.0** Сформулировать проверяемую гипотезу в виде: «При прочих равных, включение I_SIFS в UpdateNeurons даёт [улучшение метрики X на Y%] или [не ухудшает] по сравнению с отключённым I_SIFS». Метрика X — например медиана CartPole score за N эпизодов, или стабильность весов после N тиков.
- **A.0** Провести эксперимент: два конфига (I_SIFS on/off), один сид, фиксированный набор эпизодов. Зафиксировать результат в плане: гипотеза подтверждена / опровергнута / неоднозначна. При опровержении — не строить фазы C–D на «SIFS как основа ИИ» без пересмотра гипотезы.

**Риск:** если центральная идея (фрактальные пороги → нейронные пороги) не верифицируется, проект не «рушится», но фокус смещается: Genesis остаётся движком, SIFS — опциональным модулем с ясной зоной применимости.

## 5. Интеграция с sifs_agents (мультиагентная система)

В проекте уже есть LLM-агенты (sifs_agents/, MCP, claude_mcp_server и т.д. — см. CLAUDE.md). Нужно явно зафиксировать роль Genesis-мозга:

- **Вариант A — дополнение:** Genesis-мозг — подсистема «быстрых» решений (реакция, торговый сигнал, низколатентный ответ); LLM-агенты остаются для рассуждений, планов, MCP-инструментов. Связь: мозг вызывается из агента через brain_query или HTTP; агент передаёт наблюдение и получает действие.
- **Вариант B — замена в узком слое:** В части пайпов (например только торговый сигнал) Genesis полностью заменяет LLM; остальные агенты без изменений.
- **Вариант C — разделение зон:** Чётко разделить: «что решает мозг» (низкоуровневый контроль, сигналы) и «что решают агенты» (стратегия, риск, текст). Документировать в архитектуре.

**Зафиксировано:** выбран **вариант A** (мозг — дополнение к агентам). Документ: [docs/SIFS_AGENTS_INTEGRATION.md](docs/SIFS_AGENTS_INTEGRATION.md) — выбор, места реализации (brain_query, brain_http_client, конфиг fallback), варианты B/C на будущее.

## 7. Чекпоинтинг и версионирование весов

- **Минимум:** после Night Phase (или каждые N тиков) сохранять состояние весов (и при необходимости сомы) в файл с меткой времени или версией. Формат: бинарный дамп, совместимый с загрузкой в тот же шард.
- **Откат:** при деградации метрики (например CartPole score падает ниже порога) — возможность подставить веса из предыдущего чекпоинта.
- **A/B весов:** возможность запускать два набора весов (например с I_SIFS и без) и сравнивать метрики без перезапекания.

**Описание схемы:** [docs/CHECKPOINTING_WEIGHTS.md](docs/CHECKPOINTING_WEIGHTS.md) — когда сохранять, формат, откат, A/B весов. Конкретная реализация в коде — в рамках фазы B/C.

## 8. Population Coding и S-уровни

**Вопрос:** Population Coding (кодирование входа полем нейронов с рецептивными полями) в Genesis и S-уровни SIFS (0..9) — как соотносятся?

- **Вариант 1:** S-уровень задаёт **зональную** привязку (шард/зона имеет один s_level); внутри зоны по-прежнему Population Coding по входам. Тогда SIFS модулирует пороги/ток по уровню шарда, а не по отдельным нейронам входа.
- **Вариант 2:** Часть виртуальных аксонов или нейронов явно маппится на S-уровни (например, группы по 64 нейрона на уровень); тогда кодирование и масштаб связаны явно.

В плане зафиксировать выбранный вариант при внедрении S-Sharding (A.2) и отразить в конфиге/документации. Устраняет пробел «Population Coding не связан с S-уровнями».

- **Источник:** `.feather` в sifs_ft/data/binance/ (или аналог); таймфрейм и пары — в конфиге среды.
